# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Runtime

This is a Bun project (`engines.bun >= 1.3.0`). Use `bun` rather than `node`/`npm`/`npx`. Module imports use the `.js` extension on TypeScript sources (`moduleResolution: bundler`), and `@/*` resolves to `src/*`.

The findings watcher uses `fs.watch(dir, { recursive: true })`. macOS supports this natively; Linux requires Node ≥ 20.0.0. On unsupported runtimes the live `+ finding:` log lines may not fire for drafts written to subdirectories, but the audit itself completes correctly (quarantine and `summarizeFindings` read the directory directly).

## Commands

```bash
bun install
bun run dev -- run --mode lite --agent claude --target ./tests/fixtures/...   # CLI from source
bun test                                          # all tests (unit + integration)
bun test tests/unit/orchestrator.test.ts          # single file
bun test --test-name-pattern "topological"        # single test by name
bun run typecheck                                 # tsc --noEmit (strict + exactOptionalPropertyTypes)
bun run transform                                 # regenerate src/content/sdk-variants/ from source content
bun run build                                     # current-platform binary → build/dist/ + ~/.local/bin/xevon-audit
bun run build:all                                 # all 4 targets (darwin/linux × arm64/x64)
bun run release:dry / bun run release             # package + (optional) mc upload to R2
```

`bun run build` runs `transform-content.ts` and `bundle-content.ts` first — those steps regenerate `src/content/sdk-variants/` and `src/content-bundle.json` (both gitignored). If you edit anything under `src/content/agent-defs/` or `src/content/command-defs/`, dev mode picks it up immediately, but the compiled binary needs a rebuild for changes to ship.

`XEVON_AUDIT_BUILD_NO_INSTALL=1` skips the post-build copy to `~/.local/bin/`. `XEVON_AUDIT_BIN_DIR=…` overrides the destination.

## Architecture

xevon-audit is a thin orchestrator that drives Claude Code or Codex through a fixed audit methodology. The methodology itself (agent prompts, mode workflows, skills) is **vendored content** under `src/content/`, not code — the engine just parses YAML phase graphs and dispatches phases to adapters.

### Three-layer execution model

1. **CLI** (`src/cli/{run,setup,verify,uninstall,incremental-scope}.ts`) — entry points wired via `cac` in `src/index.ts`. `run` resolves auth/adapter/mode and constructs an `Orchestrator`. Default mode prints the human log to **stdout** (so `| tee log` works); only true errors, fatals, and the SIGINT abort notice go to stderr. The global `--json` flag replaces the human log with NDJSON on stdout. Chalk auto-strips colors when stdout is not a TTY.
2. **Engine** (`src/engine/`) — `Orchestrator` loads a `CommandDef` (parsed from `command-defs/<mode>.md` frontmatter), runs `topologicalOrder` over its `phases:` graph, and drives each phase through one `Adapter.run()` call. Phase state persists to `xevon-results/audit-state.json` via `StateStore`; supports resume, `requires_git` skip, transient retries with backoff, and a `findings-draft/` filesystem watcher that emits `findingDiscovered` events. Failure policy is `skip-and-continue` by default (`--strict` aborts on first failure).
3. **Adapters** (`src/adapters/`) — `Adapter` is an interface that streams a normalized `AdapterEvent` union (`textDelta` / `toolCall` / `toolResult` / `thinking` / `finish` / `error`). Four implementations: `claude-cli`, `claude-sdk`, `codex-cli`, `codex-sdk`. `chooseAdapter()` in `detect.ts` picks SDK when an API key env var is set, otherwise CLI (relying on the binary's ambient subscription auth).

### Content layering

`ContentLoader` (`src/content-loader.ts`) is the single point of truth for finding agent/command/skill/harness files. Resolution order:

1. Per-user override at `~/.config/xevon-audit/{agents,commands,skills}/<name>` (or `$XEVON_AUDIT_CONFIG_DIR`).
2. SDK-safe variant under `src/content/sdk-variants/` when `variant: "sdk"` is requested.
3. Embedded copy under `src/content/{agent-defs,command-defs,skills}/`.

The orchestrator picks `variant: "sdk"` whenever the adapter's platform is `codex` (`Orchestrator.contentVariant()`). The SDK variant is generated by `scripts/transform-content.ts` — pure deterministic regex rules that strip Claude-Code-shaped slash commands, `xevon-audit:` agent prefixes, `run_in_background: true`, etc. Regenerate after any edit to `agent-defs/` or `command-defs/`; CI doesn't do this for you.

In compiled mode the entire `src/content/` tree is inlined into `src/content-bundle.json` at build time and extracted to `~/.cache/xevon-audit/content-<hash>/` on first run (hash derived from `generated_at` so a new build invalidates the cache).

### Mode = phase graph in YAML

Each `command-defs/<mode>.md` file has frontmatter that declares the `phases:` array (id, title, agent, requires_git, parallel_with, depends_on). `engine/phase.ts` validates with Zod, detects cycles, and topologically sorts. v1 walks the order sequentially — `parallel_with` is recorded but not yet honored. Phases with `agent: null` are "inline" and run with the command-def body as the system prompt; phases with an agent name load that prompt from `agent-defs/<name>.md`.

When customizing modes or adding new ones, the engine has no hardcoded phase IDs — the YAML graph is canonical. Any phase referenced in `depends_on`/`parallel_with` must exist in `phases:`.

### Harness installation

`engine/harness.ts` and `cli/setup.ts` are separate from the runtime adapters. `setup` merges `harnesses/<platform>/frontmatter.yaml` (per-agent tool allow-lists, model assignments, exclusions) into the agent definitions and writes them to the platform's plugin/agents directory:
- Claude → `~/.config/xevon-audit/harness-claude/` (Claude Code plugin format with `.claude-plugin/plugin.json`, `agents/`, `commands/xevon-audit/`, `skills/`).
- Codex → `~/.codex/agents/xevon-audit-<name>.toml` (single-file agents with `name="xevon-audit:<agent>"`).

Interactive `xevon-audit run -i` calls `ensureInstalled(platform)` automatically before exec'ing the underlying CLI. Headless runs don't need the harness install — they drive the SDK/CLI directly.

### Auth overrides

`engine/auth-overrides.ts` applies `--api-key`, `--oauth-token`, `--oauth-cred-file` for the lifetime of one run and restores the original state on exit (including SIGINT/SIGTERM). The cred-file path moves the original file to `<target>.xevon-audit-backup` and writes the override in place; the restore handler is registered before any adapter work begins. When adding new auth surfaces, plumb them through `applyAuthOverrides()` rather than mutating `process.env` directly so the restore path stays correct.

### Audit output

Audit artifacts land in `<targetDir>/xevon-results/`:
- `audit-state.json` — phase graph state (canonical for resume, diff baseline, merge sources).
- `findings-draft/<phase-id>-<slug>/` — in-progress findings (watched by orchestrator).
- `findings/<Severity><N>-<slug>/` — finalized findings (must match `<Severity><N>-<slug>` against frontmatter).
- `final-audit-report.md`, `confirmation-report.md`, `merge-report.md`, etc. depending on mode.

The orchestrator's `quarantinePartialOutput` moves drafts of failed phases to `xevon-results/.archive/<auditId>/<phaseId>/` so retries don't merge against half-written files.

## TypeScript conventions

`tsconfig.json` enables `strict`, `noUncheckedIndexedAccess`, `noImplicitOverride`, and `exactOptionalPropertyTypes`. `exactOptionalPropertyTypes` in particular changes how optional fields work — you must use conditional spread (`...(opt !== undefined ? { field: opt } : {})`) when constructing objects with optional fields rather than passing `undefined` directly. There are many examples of this pattern in `orchestrator.ts` and `run.ts`.

## Testing notes

- `tests/integration/e2e-lite.test.ts` uses a `ScriptedFakeAdapter` that simulates a phase-by-phase run by writing real files to a tmpdir. Use this pattern for new e2e tests rather than mocking the SDK.
- Unit tests cover individual engine modules; they never spawn `claude`/`codex` and never hit the network.
- `bun test` reads from `src/content/` directly (dev mode), so make sure your local `sdk-variants/` is up to date if a test exercises the codex/SDK code path.
